## [TRAVEL-001] ✅ 已修复（2026-06-11）病灶根①：断裂 C 转换契约不完整 + type 词表三套不统一（P0 · 最致命 · 横扫稻城/台湾/泰国）

# 第二批压测立项（2026-06-12 · 病灶 ⑥~⑭ · 全部待修）

> 一句话诊断：前 5 个病灶是「管道内部会丢东西 / 判断打架」；第二批暴露的是更深的共性短板——**skill 很会「把东西摆整齐」（承载位 / 段数 / region 匹配都校验得细），但几乎不「核对这些东西本身对不对」**（季节 / 地理 / 位置 / 资格 / 诉求矛盾全靠 Claude 自觉，无门禁强制）。

## 元根因（评审归并 4 条）
- **元根因 A · 重承载位、轻事实核查**——所有核查门都在「数据写完后」查结构完整性，没有一道在「动手前」查前提本身是否符合事实。覆盖 ⑥⑪⑫，部分 ⑦。**最该优先补。**
- **元根因 B · 只有局部校验（当天级/跨城级），缺整趟级宏观可行性门**——矛盾诉求不摊牌、硬排或静默降级。覆盖 ⑦，部分 ⑥⑩。**最该优先补。**
- **元根因 C · 约束体系是「负向把关 lens + 单值档」**——撑不住正向主题 lens / 混合分项弹性预算 / 会改变筛选强度的同行结构。覆盖 ⑨⑩⑬。多数可规则层补强、零界面。
- **元根因 D · 几何模型是「离散城市 + 公共交通跳转」**——撑不起自驾连续路径 / 移动即内容。覆盖 ⑧，部分 ⑫。唯一伤筋动骨、命中窄，缓做。

## [TRAVEL-007] ✅ 已修复（2026-06-12）· 病灶根⑥：全链路无「事实自洽性」核查门（P0 · 4 案例全中 · 元根因 A · 第一波）
**症状**：skill 信任用户给的目的地集合 / 玩法绑定 / 季节常识本身成立，整条管道只在「数据写完后」做结构核查，从不在「动手前」核查前提是否符合地理/季节/位置事实。
- 案例1：腾冲在云南（亚热带不下雪）被当东北第三城、「腾冲玩冰」伪需求被当真；断裂 C「吉林→腾冲」3000km 反常识城际段，9.2 城际对账只数段数+比字面 from/to，照样绿灯通过。
- 案例2：阿勒泰（最北）与像样沙漠营地（东南/南疆）对角两端 1000km+ 不相容，无整趟级地理可行性预检。
- 案例3：影视取景地真实位置≠用户预期城市（《苦尽柑来》釜山桥段实拍在陕川、油菜花田在高敞，都不在三城内），用户去釜山扑空；9.1 只在 region 标错时抓，抓不到「取景地超出所有计划城市」。
