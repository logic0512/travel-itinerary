## 三、工程规则

### W-1 交付节奏（金标准）

| 数据量 | 交付方式 |
|---|---|
| 1-5 张 cards | 一次做完 |
| **6-15 张** | **先 1 张样板 → 用户验收 → 1 天 1 天交付** |
| **16+ 张** | **T1 骨架 → T2 核心数据 → T3 cards 1 天 1 天** |

**铁律**：用户主动建议"按 1 天 / 3 天拆"是金标准·不要赶。

### W-2 写数据前必读渲染函数

进 Stage 3 第一件事·读完：
- `renderCard()`
- `renderTimelineItem()`
- `renderDay()`
- `renderDayMapThumb()`
- `collectDayCoords()`
- `renderStickyCtx()`
- `renderQuickCards()`

把它们消费的所有字段列成 schema·**作为 cards 数据的写作模板**。

> ⚠️ **schema 注释 ≠ 渲染函数真实消费的字段**。模板里 `const DAYS=[ // ... ]` 的注释可能过时/不全，
> **以 renderDay/renderDays/renderTimeline 实际读的字段为准**。台湾 case 踩过两次：
> ① 注释没写 `num`，但渲染用它做容器 id；② 注释写 `reminders/highlights`，渲染读的是 `day_todos`。

### W-2.1 ⭐ DAY 必含 `num`（v2.4.2 血泪规则）

每个 day 对象**必须有唯一的 `num`（1 起）**。渲染函数用它生成容器 id：Q2 是 `tl-d${num}`/`day-${num}`，
trip 是 `day-${num}`/`daymap-${num}`。

**漏填 `num` 的后果（灾难级，且不报错）**：`d.num===undefined` → 所有天容器 id 都变成 `tl-dundefined` →
6 天时间轴全部渲染进同一个（第一天的）容器，最后一天的内容盖在最上面。**症状：D1 标题配 D6 内容、标题显示「DAY undefined」**。

**v2.4.2 已在两个模板加护栏**（缺 num 自动按顺序补、reminders/highlights 自动归并进 day_todos），
但仍应**显式填 `num` + `day_todos`**，别依赖兜底。填完用下面的自检脚本验证：

```bash
# 抽出数据段，模拟 num 赋值，断言容器 id 唯一 + 每天时间轴归属正确
node -e '...DAYS.forEach((d,i)=>d.num=d.num??i+1); 断言 new Set(DAYS.map(d=>"tl-d"+d.num)).size===DAYS.length'
```

### W-3 内联 trip-data 优先（双击即用）

```html
<!-- 在主 script 之前 -->
<script id="trip-data" type="application/json">
{...完整 trip.json...}
</script>

<script>
async function init() {
  const inlineEl = document.getElementById('trip-data');
  if (inlineEl && inlineEl.textContent.trim().length > 10) {
    TRIP = JSON.parse(inlineEl.textContent);
  } else {
    TRIP = await fetch('trip.json').then(r => r.json());  // 兜底
```

### W-4 inline JSON 注入用 str.replace（不用 regex）

```python
# ❌ 错（regex lazy 匹配会从注释里的字符串扫起）
content = re.sub(r'<script id="trip-data"...>.*?</script>', new_block, content, flags=re.DOTALL)

# ✅ 对（字符串替换）
old_block = content[content.index('<script id="trip-data"'):content.index('</script>', ...)+9]
content = content.replace(old_block, new_block)

# ✅ 也对（regex 用 lambda·避免反向引用问题）
content = pattern.sub(lambda m: new_block, content)
```

## 四、事实核查清单

### I-0 输入物识别清单（v2.6.1 新增）

用户给文件/截图时，先判断类型，再开始规划：

| 输入类型 | 识别动作 | 不可犯错 |
|---|---|---|
| 手机备忘录截图 | 抽取 Day、地点/食物、顺序、偏好词 | 不要把口语食物名直接当餐厅实体 |
| 手绘路线图 PDF/图片 | 抽取 day/time/place/transport/note/locked/needs_recheck | 不要推翻用户路线；先校正事实和时间 |
| 图片型 PDF | 先渲染成页面图或 OCR，再读内容 | 不要只跑 `pdftotext` 后认为无内容 |
| 票据/订单截图 | 抽取出发/到达/票号/酒店/取车/还车时间 | 不要把票据时间当可变项 |
| 地图收藏/POI 列表 | 去重、分片区、分必去/可选/待查 | 不要直接按列表顺序排路线 |

执行步骤：

1. 输出 `artifact_summary`：已识别内容、缺失内容、疑似风险。
2. 给每条来自文件的原始信息打 `source=user_artifact`。
3. 对路线图中的每个节点建立结构化草表：`day / time / raw_label / normalized_place / transport / confidence / needs_recheck`。
4. 只有经过 web_search 或用户确认后的节点，才能升级为 timeline 的确定项。

### I-1 已有偏好流转：消歧 → 预填 → 去重（v3.0 · 防丢/重/甩回）

用户给的「半成品」（口语条目、模糊指代、手绘潦草地名）必须走完整条流转链，任一环节都不准弄丢、弄重、或把本该系统查的甩回让用户确认。

**① 消歧（断裂 0 · 进任何界面前必做）**
- 口语/模糊指代——「那家很辣的螺蛳粉」「那个很出名的大象公园」「手绘上潦草的地名」——一律先 web_search 锁定到具体 POI（店名 + 地址 + coords）。
- **可直接锁定的唯一条件（量化 · 关掉心证口子）**：web_search 结果**高度指向唯一一家**具名实体。判定法：**搜索结果里任意两条指向不同实体（不同店名/地址）= 有候选冲突**（目录页/新闻/榜单不计为候选实体）；无冲突才可直接锁定。只要出现 **≥2 个合理候选**，或修饰词（「最辣」「很出名」）不唯一指向某家 → **一律按一对多歧义处理，不准自行拍板锁定**。
- **一对多歧义**：列最可能的 1–2 个候选写进 `artifact_summary` / 分诊声明，**回钩用户一眼确认**（候选由 web_search 给出，每个附一句区别理由，如「A=禁骑象保护区 / B=骑象表演营」）。
- 消歧结果进 fact_ledger：`raw_label → resolved_poi（或候选列表）+ confidence + needs_user_confirm`。
- 查不到 → 标 `needs_recheck`，作为备选不进硬 timeline。

**② 预填（进 discovery / planner 时）**
- **已锁定到唯一 POI** 的用户已点名条目 = partial 维已有锚点，必须顶在前面：走 A 在 discovery **预勾选 ★必去**（不混进待发现候选让用户重找）；走 B 直接作为 planner 的 locked/partial 锚点注入。
- **消歧未锁定唯一（多候选）**：**不预勾选★**，标为「待确认候选」（区别于普通发现候选），用户选定后再升★——不准默认全标★让用户以为都要去。
- **半具名方向词**（有类型无具名点，如「想多看看古镇」「吃点地道的」）：归 **open 研究候选**，进 discovery 标「偏好方向」而非 ★必去，交给 Claude 主动研究出具体候选，不算用户已点名。
- open 维（Claude 主动研究的**新增**候选）与用户已点名的分开标，不准盖掉用户的。

**③ 去重（断裂 C 合并 TRIP 时 · 不靠死匹配，靠 Claude 语义比对 + 清单核查）**
- planner 与 supplement 是**两个解耦界面、各自生成 id**，同一家店两边 id 必然不同、名字也可能写法不一（「螺蛳粉X」vs「X螺蛳粉」）——**严禁只靠 id/字符串精确匹配去重**（必漏）。
- 正确做法：合并 `extras` 前，Claude 把 **planner timeline 已有点位 × supplement 备选**做一次**语义比对**（认得出同一家店的不同写法/不同 id），输出一份「以下 N 项已在行程内、从 extras 移除」的去重清单作为合并日志。
- 验收：同一家店不得在 timeline 和 extras 同时出现；去重清单要能被用户一眼核对。

### F-0 事实台账（v2.6 新增）

每个 case 都必须复制 `references/fact_ledger.template.md` 到本次 case 输出目录，命名为 `fact_ledger.md`。如果用户不需要单独文件，也要在分析报告中保留同格式事实台账。

| 字段 | 说明 |
|---|---|
| fact | 具体事实，如「某餐厅周三休」 |
| source | 来源名称 + URL 或搜索结果标题 |
| checked_at | 查询日期，格式 YYYY-MM-DD |
| confidence | high / medium / low |
| impact | 对行程影响：critical / normal / FYI |
| action | adopt / avoid / needs_recheck / fallback |

硬规则：

- high：官网、官方地图、官方社媒、交通官网等直接来源。
- medium：大型平台、订票平台、餐饮平台、近期多源一致。
- low：博客、攻略帖、旧内容、单一非官方来源。
- critical + low 不得写进不可改时间轴，只能写进提醒或备选。
- 两个来源冲突时，写明采用哪个，另一个作为风险提醒。
- 所有用户提供的地点/班次/营业信息，都要进入台账并二次核实。

### F-1 餐厅 4 必查

每个餐厅写卡前用 web_search 验证：

1. **定休日**（特别看是否撞行程日）
2. **营业时间**（开门 / 关门 / L.O.）
3. **是否接受预约**
4. **末班 LO 时间**（影响晚饭安排）

### F-2 用户特定地名必查

用户说"X 站去 Y 寺"·"小渕町坐车去 X"等具体动线时：
- 全部 web_search 验证
- 不要靠常识猜
- 巴士 / 电车具体班次也要查

### F-3 关键店铺验证清单（标准）

```
店名 + 地址 + 营业时间 + 定休日 + 价位 + 历史
```

每张 Tier A 卡至少 1 次 web_search·关键店（已订位的）至少 2 次（一次查时间·一次查菜单/历史）。

## 五、Stage 3 执行流程（推荐）

```
1. 读 references/pdf-output-rules.md，确认 HTML-first / browser-export PDF 管线
2. 列出 cards schema（按本文档第一章）
2b. 复制 fact_ledger.template.md → fact_ledger.md，开始填台账
2c. 运行 fact_ledger.md 的 Pre-Stage 3 Gate——四项全部通过才继续
2d. 复制 asset_ledger.template.md → asset_ledger.md，收集每个非 transit 行程点的 image_asset
2e. 写 visual_brief（格式见 design-rules.md 第二章），确定 HTML 画册方向
3. T1 HTML 画册骨架（trip-print.html：cover / overview / daily page / checklist）
4. T2 meta + days timeline 主轴 + 每日视觉锚点
5. T3 cards · 第一天作为样板（5-12 张完整字段 + 图片）
   ↓ 用户验收
6. T3 cards · D2 D3 D4 D5 1 天 1 天交付
   ↓ 每天验收
7. T4 浏览器打开 HTML → 截图验收 → 导出 trip-guide.pdf + todo-before-trip.md
8. PDF 逐页截图验收 / 复盘
```

### Stage 3 禁止路径（v2.8 必读）

以下做法一律不合格，即使生成了 `trip-guide.pdf` 也要重做：

- 用 ReportLab / FPDF / PDFKit / LaTeX 直接画最终 PDF。
- 只有 PDF，没有 `trip-print.html` 或 `trip.html` 源文件。
- PDF 里出现 `口口口`、emoji 方框、中文字体 fallback 错乱。
- 用渐变色块、黑块、空框、emoji 当“图片”。
- HTML 还没截图验收，就直接宣布 PDF 完成。

## 九、v2.5 新增核查规则（台湾 solo case 血泪）

> 以下 6 类规则均为 **P0**——任一项失败视为 bug，必须修完再往下走。

---

### 9.0 路径分诊门（开工前必做 · 最高优先级）

> **9.0 门是整个 skill 仅有的"开工前"检查点，含两道开工前关：事实预检（9.0.0）+ 路径分诊（9.0.1 起）。其余 9.1–9.6 都是"数据写完后"的核查。**
> **两关有先后**：事实预检（9.0.0）在最前——前提错了（季节填反 / 地名错省 / 玩法绑错地点 / 取景地散在异地），分诊和下游再准也全白搭；前提核对通过，才进路径分诊。
> 路径选错 = 下游全错（错的界面组合 / 错的研究力度 / 错的交付物），且错误隐蔽——产出可能看着完整，实则漏掉目的地精华或逼用户重做已定项。
> **未过事实预检（9.0.0）、未输出分诊声明（9.0.1），不得进入任何 Stage 0。**

#### 9.0.0 事实自洽性预检门（病灶⑥修复 · 开工前第一关，先于分诊声明）

> **病灶⑥修复**：skill 过去只在"数据写完后"查结构（region 匹配 / 坐标密度 / 城际段数都校验得很细），却从不在"动手前"查用户给的**前提**对不对——季节、地理、玩法可行性、具名实体真实位置全靠 Claude 自觉、毫无门禁。结果硬伤（腾冲被当东北城、南半球元旦填成冬天、釜山取景地实拍在陕川）一路绿灯产出错误攻略。本节强制在动手前先过一道事实关。

进任何 Stage 0 / 分诊（9.0.1）之前，**先带着 4 类高频陷阱意识把用户给的前提扫一遍**，命中疑点必须 WebSearch 核实，核实后才能进 9.0.1 分诊：

**四个必查项（命中疑点必须 WebSearch 核实）：**

1. **目的地集合地理自洽**：多目的地能否合理串成一条动线？有没有离群项（跨大区 / 跨气候带 / 相距上千公里）？例：腾冲（云南）混进黑龙江 + 吉林 = 离群。命中 → 回钩「你说的腾冲其实在云南、离黑/吉约 3000km，是不是想去漠河 / 雪乡这类东北雪点？」
2. **季节 × 出发地半球**：目的地在南半球、或当月季节可能反直觉时，**必须 WebSearch 核实当月真实气候，禁止按出发地 / 北半球常识推断**。例：澳洲元旦 = 仲夏 35℃，季节约束该写防晒 / 避正午，而不是按"元旦=冬"填保暖。核实结果**直接用于修正 season 约束方向**。
3. **玩法 × 目的地可行性**：把用户每个玩法诉求绑到具体目的地核实——这地方这季节能不能做这件事？例：腾冲不下雪 →「腾冲玩冰」不成立。命中 → 回钩纠正（把"玩冰"落到哈尔滨 / 吉林）。
4. **具名实体真实位置 / 可得性**：用户点名的具名点（景点 / 影视取景地 / 特定餐食 / 特产）核实真实所在地与可得性，若落在计划城市 / 区域外、或本地稀缺 → 回钩（加点 / 专程一日游 / 放弃 / 告知稀缺）。**用户提供的剧名 / 店名 / 专名可能记错，先核实规范名称再查取景地 / 位置**（例：用户说"苦尽甘来"实为《苦尽柑来》；《苦尽柑来》釜山桥段实拍在陕川、不在釜山；鳄鱼肉在悉尼属猎奇菜、需提前确认可得）。

**轻重控制（必守 · 防过度设计）**：预检**不是**对每个地名都 WebSearch，而是"带着 4 类高频陷阱意识扫一遍，命中疑点才深入核实"。高频陷阱清单：① 季节反直觉（南半球 / 反季目的地）② 地名误植（同名 / 记错省份 / 口误）③ 玩法-地点错配 ④ 取景地 / 特色物散落异地或本地稀缺。无疑点的常规目的地，预检一行带过即可，不拖慢正常 case。

**产出与回钩**：发现矛盾 → **不静默修正、也不闷头照做**，列「用户说的 vs 核实到的」回钩用户一眼确认（复用 §I-1 消歧回钩机制：列候选 + 一句区别理由，等用户拍板）。预检通过（无疑点或已澄清）才进 9.0.1 分诊。

**固定输出格式（在分诊声明之前先输出）：**

```
【事实预检】（开工前 · 先于分诊）
- 地理自洽：✓ / ⚠ 离群目的地 ____（核实：__在__，距其他目的地约__km）→ 请确认是否__
- 季节核实：____（目的地当月真实气候；南半球/反季必查，禁按常识推断）
- 玩法可行：✓ / ⚠ ____在____不成立（____）→ 建议改为____
- 具名点位置：✓ / ⚠ ____实际在____（不在计划的____）→ 加点 / 专程 / 放弃？
```

无疑点则四项全 ✓ 一行带过。

**与其他核查的分工**：9.0.0 是"动手前"查**前提**（季节 / 地理 / 玩法 / 具名点位置对不对）；F-0 事实台账 / 9.2 是"数据写完后"查**结构**（坐标 / 段数 / 营业状态）。两者互补、不重复。

#### 9.0.1 强制分诊声明（第一句话，固定格式）

进入 Stage 0 前，**必须先对用户输出**以下声明，让用户能一眼否决：

```
【路径分诊】
- 目的地：____（已定 / 未定）
- 输入物：____（无 / 手绘PDF→C-lite / 备忘录截图→B / 票据→锁定项 / POI清单→B）
- 同行约束：____（人数 / 适老 / 儿童 / solo —— 影响所有 open 段候选筛选）【未提供时，列入"最多 3 个必问"强制追问，不得默认成人直接排】
- 认知深度：低 / 中 / 高 —— 一句话理由（知道去哪 ≠ 知道怎么玩）
- 已锁定段：____（机票/酒店/门票/必去核心/区域顺序，逐项列）
- 待填段：____（哪些天、哪些城市内细节、吃喝是空的）
→ 主路径判定：A / B / C —— 按下方两步法（先看骨架定主路径，有骨架再用占比缺口分 B/C）
→ 若你其实已锁定更多（每天都排好了），回我一句，我切到更轻的路径
```

**主路径取法（两步法 · 流程看骨架、力度看占比，各用各的尺子）**

A/B/C 的本质区别是「要不要走 discovery 帮用户从头发现景点」。判据是**用户有没有自带行程骨架**，**不是**完成度百分比——完成度低 ≠ 需要发现（用户可能路线想好了，只是没核实/没订）。

**第 1 步 · 看骨架定主路径（用 D4 认知深度，数数不靠感觉）**
- **无骨架 → A**：只有目的地+天数；或只有方向词无具名点；或多城但「有具名点的城市数 ÷ 总城市数 < 半数」。
- **有骨架 → 进第 2 步**：用户自带以下任一——区域/城市顺序框架；或 ≥1 个主动指定的必去具名点（说得出为何去/优先级）；或手绘逐日动线。

**第 2 步 · 有骨架者用占比缺口分 B / C**
- 缺口 **≥ 20%**（还有不少要填/研究）→ **B**：跳过 discovery，planner 载入骨架 + Claude 补研究的点。
- 缺口 **< 20%**（5 维基本全 locked、已订或已核实）→ **C**：仅核实排序。

**占比缺口怎么算（5 维度计数，消除心证 · 结果只用于第 2 步的 B/C 分界 + 9.0.2 分段力度，不单独决定 A）**：把结构拆成 **5 个维度**，逐维判 locked / partial / open：

| # | 结构维度 | 判 locked 的条件 |
|---|---|---|
| 1 | 景点/体验编排 | 具名点已定且已核实开放 |
| 2 | 每日动线/顺序 | 每天先去哪后去哪已排 |
| 3 | 餐饮 | 具名店已定且已核实营业 |
| 4 | 交通（城内+跨城） | 班次/方式已定 |
| 5 | 住宿落点 | 已订或已定区域 |

**缺口 = (open 维度数 + 0.5 × partial 维度数) / 5。**
- **单维 / 多维饱和都被它覆盖**：饱和维标 partial（逐个核实+排序），未覆盖维标 open（主动研究）。

**冲突裁决（D1 · 最重要）**：占比缺口很高（如 >60%）但用户有完整骨架（手绘逐日 + 必去点 + 区域顺序）时——**主路径听骨架（走 B），高缺口不把流程拉回 A，而是转成「open 段大力研究填补」的力度要求**（见 9.0.2）。一句话：**骨架决定走哪几个界面，占比缺口决定每段花多少力气，两把尺子各管各的，谁也不覆盖谁。** SKILL.md 的「信号表」「输入物表默认路径」只给起点猜测，与本两步法冲突时一律以本法为准。

**认知深度硬锚点（D4 · 即「有无骨架」的操作化判据，消除自由心证）**：
- **高（有骨架）**：给了每天大致安排 / 手绘逐日动线 → 第 1 步过，按缺口分 B/C。
- **中（有骨架）**：≥1 个具名必去点 **且** 说得出为何去（明确优先级）→ 第 1 步过，按缺口分 B/C。
- **混合情形**：部分点说得出为何去、部分说不清 → 以"说得出为何去"的点作**骨架锚点**（有骨架，进 B/C），说不清的点标 open/partial 按力度层研究，**不因部分点模糊就整体打回 A**；唯有"全部说不清"才落 9.0.3「报了一串地名」陷阱 → 无骨架走 A。
- **低偏中（无骨架）**：只有方向词（"看古建""吃好吃的"）无具名点 → **A**，先研究出候选再让用户确认。
- **低（无骨架）**：只给目的地+天数，或明说"不知道玩什么/你帮我定" → **A**。
- **多城（无骨架判据）**：有具名点的城市数 ÷ 总城市数 **≤ 半数** → 视为无骨架走 **A**（正好半数也走 A——空白城占一半及以上，让它走完整 discovery 由用户参与选点）；**> 半数**才算有骨架，进 B/C。例：泰国 3 城只清迈 1 个具名点（1/3 ≤ 半数）→ A，不准因"有 1 个具名点"判中→误走 B/C。
- **典型双重确认**：备忘录列满想吃的店但没景点/区域骨架 → 第 1 步无骨架 → **A**；占比缺口 (4 + 0.5)/5 = 90% 也指向"全程大力研究"。**绝不因"每天都有条目"判成 B/C。**

#### 9.0.2 分段成熟度表（结构层 · 承认一趟行程成熟度不均）

一趟行程不是单一成熟度。开工时必须把行程**按城市/按天段**标出成熟度，研究力度按段分配：

| 成熟度 | 判据 | skill 动作 | 对应逻辑 |
|---|---|---|---|
| `locked` | 已订票/已订酒店/用户明确必去且不可改 | **只核实**（开放状态/时间/交通衔接），不重排、不推翻 | C 逻辑 |
| `partial` | 给了方向或几个点，其余空 | 核实已有点 + **主动研究填补**空缺 | B 逻辑 |
| `open` | 整天/整段空白，用户无具体想法 | **主动调研 + 带用户发现**（discovery 界面让用户勾选），禁止甩给用户自己填 | A 逻辑 |

- 把每个 city / day 标到上表，写进分析报告。
- **铁律**：`locked` 段不浪费调研力气去推翻；`open` 段绝不偷懒跳过主动研究。
- 主路径（9.0.1）只决定**走哪几个界面**（A=discovery起 / B=跳discovery / C=跳discovery+supplement）；研究力度永远按本表分段执行。

**强制开放段校验（D2 · 防主路径吞掉分段力度，A/B/C 对称）**：
- [ ] **主路径 = C/C-lite**：本表若存在**任一** `open`/`partial` 段 → **必须显式列出**并写明"将主动研究填补"。最常见漏点：机酒动线全订（看着纯 C）但**没提吃喝** → 餐饮恒 `open`，C 同样不得跳过三餐。
- [ ] **主路径 = B**：若用户偏好只覆盖了**一个维度**（如只有餐饮、只有景点）→ **其余所有维度必须显式列为 `open` 并主动研究**，不准因"有内容"就停止补全。`locked` 段不得重排。
- [ ] **主路径 = A**：全程主动研究，但已有的零星具名点（如唯一一个必去点）标 `partial` 只核实不推翻，不得连用户明确想去的也一起当空白重排。
- 一句话门禁：**主路径决定走哪几个界面，本表决定每段花多少力气——两者各管各的，任何路径都不准用主路径一刀切覆盖分段力度。**

#### 9.0.3 防误判自检 3 问（判定前逐条过）

- [ ] **「只给目的地+天数」陷阱**：用户只说"我要去 X 玩 N 天"、没有任何具体景点/体验偏好？→ 这是 `experience_knowledge 低`，必须走 A（主动研究），**不准**因为目的地明确就判 B/C。
- [ ] **「报了一串地名」陷阱**：用户报了几个地名，但说不清为什么去、去那儿干嘛、哪个优先？→ 这些是 `open/partial`，不是 `locked`；不准当成"行程已定"跳过研究。
- [ ] **「给了笼统方向」陷阱**：用户给了方向词（"看古建""吃好吃的"）但无具体点？→ 认知深度判低偏中，先研究出候选再让用户确认方向，**不准**直接拿方向词排 timeline。
- [ ] **「框架覆盖率不足」陷阱（D3）**：用户给了手绘/每日安排（=有骨架，主路径至少 B），但只覆盖了部分天（如 7 天只画了 3 天）？→ 已画的天按 `locked/partial`、**未覆盖的天一律按 `open`**（力度层）；既不准因"有手绘"就判 C（除非 5 维基本全 locked），**也不准因"吃住空、缺口高"把它打回 A**——主路径听骨架（见 9.0.1 两步法的冲突裁决）。
- [ ] **「信息过载需做减法」陷阱（D3）**：必去点/愿望明显超出天数容量（如 7 个硬点塞 2 天）？→ 真正待办是**先做减法对齐容量**，不是硬排满；先跟用户砍/挪，再排 timeline。**多城/大跨度行程不止算点位密度**：还要估「城际交通总耗时占总天数比例」，超阈值（**>30%**）同样触发减城/减点 → 移交 9.0.7 整趟可行性摊牌门处理（列冲突 + 2 个取舍方案回钩用户裁决）。

任一陷阱命中而路径判轻了（B 当 C、A 当 B），即为误判，重判；过载陷阱命中则先做减法再排。

#### 9.0.4 数据落地

**必须写进结构的字段（缺一不可）：**

- `meta.route_path`（'A'|'B'|'C'，记录主路径判定结果）。
- 每个 `day.maturity`（'locked'|'partial'|'open'），驱动渲染与研究力度；缺省按主路径：**A 路径新建 day 缺省 `open`，B/C 缺省 `partial`**（A 是全程主动研究，统一缺省 partial 会让真·空白天被当成只需核实）。
- `meta.budget_tier`（'backpacker'|'standard'|'comfort'，预算档位，见 9.0.5）。
- `meta.constraints[]`（贯穿约束**减分** lens 清单，结构 `{type, text, scope}`，见 9.0.5）。
- `meta.themes[]`（贯穿主题 / 朝圣**加分** lens，结构 `{name, kind, anchors[], scope}`；用户有贯穿性主题主线时才有，见 9.0.5⑥）。
- `meta.city_plan`（多城 ≥2 城才有；城际骨架 `{cities[], intercity[]}`，见 9.0.6）。

#### 9.0.5 开局固化约束清单（贯穿性强约束的承载位）

> **病灶④修复**：稻城高反/梅雨备雨/包车vs班车、台湾背包客省钱/独行安全这类「整趟硬规矩」，过去直到最终渲染才被手填进 `days[].reminders` 当展示文案，规划阶段完全用不上。本节强制在 Stage 0 把它们结构化固化，全程当筛选 lens。

**① Stage 0 末尾必做 · 结构化捕获**

把对话 / web_search 里散落的贯穿约束，结构化写进 `meta.constraints`，按 `health`/`safety`/`budget`/`season`/`transport`/`other` 归类，每条一句话（既是筛选依据，也是给用户的提醒）。例：

```
meta.constraints: [
  { type:'health',    text:'稻城海拔 3700m+，行程前两天低强度、备红景天/葡萄糖', scope:'trip' },
  { type:'season',    text:'梅雨季，每个户外日都要备齐室内替代点', scope:'trip' },
  { type:'transport', text:'稻城景区禁私家车，核心段走景区观光车换乘', scope:'trip' }
]
```

**② 全程作为筛选 lens（核心 · 三处都要拿这张清单对照）**

- **① discovery 选候选时**：省钱→平价候选优先；高反→海拔友好候选优先；梅雨→室内备选齐全。约束筛掉不达标候选，不是排完再补救。⚠ **这条「筛掉不达标候选」只对本节 ①～⑤ 的减分约束 + `budget_tier` 生效**：`meta.themes`（⑥）是**加分** lens，**永不用于筛除任何候选**——主题只把相关候选顶到前排、把 anchor 当锚点保入，绝不因主题砍掉非主题的吃 / 住 / 行候选。别把「主题也是一条 constraint」与本句拼起来误删点位。
- **② planner 注入排点时**：节奏/路线按约束排（高反日低强度、雨季把户外往天气窗挪、班车日卡发车时刻）。
- **③ 断裂 C 组装 TRIP 时**：**全部约束统一进「出行须知」区**（不论 `scope`），region/day 级由渲染层自动标注适用范围（`day:N`→`【第N天】`、region→`【region名】`，无中文名映射时降级显示 slug 原文）。落 `day.reminders` 改为**可选**的当天上下文增强、**非唯一出口**——「出行须知」区才是保证一条不丢的主出口。

**③ budget_tier · 预算档位（用户提了才厘清，没提不强制）+ 口径厘清（病灶⑩修复）**

预算档位（背包客 / 标准 / 舒适）写进 `meta.budget_tier`，驱动研究方向。例：`budget_tier:'backpacker'` → 餐饮查大众平价店、住宿查青旅/民宿、交通优先班车而非包车。`meta.budget_tier` 与展示用字符串 `meta.budget` 是两回事，别混。

**⚠ 用户全程没提预算 = 不强制追问（必守 · 病灶⑩补充）**：预算不是每趟都得用户拍板的输入。若从头到尾用户没主动提预算 / 档位、也没问起，**不要为了填 `budget_tier` 而强行发问，更不要拦着不让往下走**——默认按 `standard` 标准档推进，一句话标明是默认假设（如「先按标准档研究，想更省 / 更舒适随时说」），把要不要调留给用户。只有用户**主动**提了预算（给档位词或金额），才进入下面的口径厘清 / 承载流程。

**⚠ 拿到预算数字先厘清口径（必做 · 否则判错档、全盘研究方向偏）**：用户给的金额必须先确认三件事再分类——① **这是什么的钱**：全程总额 / 人均 / 还是某项专项（购物 / 化妆品 / 单项体验）？② **币种**；③ **含不含大头**（机票 / 酒店是否已另算）。不确认就套档 = 易判错。反例：案例3「5000」其实是两人化妆品专项，被误当全程总预算 → 判成背包档、餐饮住宿研究方向全偏。**专项金额绝不并进全程档位判定，单独记。**

**轻重控制（必守 · 防过度盘问）**：只在「**给了金额且口径有歧义**」时才厘清。用户已说清的（明确全程 / 人均 + 币种已知 + 说了含不含机酒），或只给档位词没给金额的（如「标准档」「越省越好」「预算充足」），**直接判档、一句话回述确认即可，不要逐条反问①②③**。只给某项专项、没给全程预算时，**不拿专项金额倒推全程档**——全程 `budget_tier` 按上面「没提预算不强制」口径处理（用户没进一步表态就默认标准档，别为补全程档硬追问）。

**现实预算常不是单一档 · 三种形态都要接住**（主导档写 `meta.budget_tier`，细节落 `meta.constraints`，`type:'budget'`，全程当筛选 lens + 进出行须知，不丢）：
- **混合比例**：如「一半住青旅、餐饮标准档」→ `budget_tier` 记主导档，混合细节落约束：`{type:'budget', text:'住宿一半青旅一半标准、餐饮标准档', scope:'trip'}`。
- **分项专项**：如「化妆品预留 5000、其余背包档」→ 专项落约束：`{type:'budget', text:'购物专项约5000，不挤占吃住交通预算', scope:'trip'}`，**专项不并进全程档**。
- **弹性降档 / 超支**：如「必要时降背包档」「为点名必去项可超支」→ 落约束：`{type:'budget', text:'弹性：常规标准档，为★必去项可超支、紧张时降背包档', scope:'trip'}`。

**与 9.0.7 轴2 衔接**：当 `budget_tier`（或弹性条款）会筛掉用户**原话点名的高价★必去项**时，**不静默砍**——交 9.0.7 整趟可行性摊牌门轴2 回钩用户（保留超支 / 换平价替代二选一）。例：案例2 1w5 背包档 vs 点名「沙漠野奢营地」。若弹性 / 专项条款是 9.0.7 摊牌门**过后**才由用户补充的，就该条款单独回补一次轴2 检查（是否新筛掉了某个★必去），不因已过门就跳过。

**④ solo / 同行约束落地（补 9.0.1 口头分诊的字段缺口）**

9.0.1 分诊声明里口头声明的 solo / 适老 / 儿童约束，过去只活在声明里、9.0.4 没给字段。现强制：除了影响候选筛选，还要写进 `meta.constraints`（`type:'safety'`），按 region / day 标 `scope`——如「花莲/台北夜间避偏僻路段、car return 前回酒店」`scope` 填对应 region（`'hualien'` / `'taipei'`）。这些约束**统一由「出行须知」区渲染并自动标注适用范围**（如`【hualien】`），保证不丢；可选额外落进该 region 当天的 `day.reminders` 作增强，但不是承载约束的唯一出口。

**⑤ 同行结构 = 一等筛选 lens（家庭 / 带娃 / 适老 · 病灶⑨半解 · 守零界面）**

> **病灶⑨**：9.0.1 口头问了「儿童 / 适老」，但过去只渲染成「注意保暖」这类一句话提醒，**驱动不了候选筛选**——同行结构是会改变行程强度的一等输入，不是展示文案。

把「带娃 / 家庭 / 适老」这类**会改变行程强度的同行结构**写进 `meta.constraints`，type 用 `'family'`（与 solo 夜间安全的 `'safety'` 区分，便于按「降强度」分支驱动；`family` 当前经「出行须知」的 其他 / 📌 兜底渲染、不丢；专属分组（模板里给 `CSTR_TYPE` 加 `family` 键 + `CSTR_ORDER` 补位，**纯模板改动**）与 selectPagesForDay 感知属缓做）。**关键是它必须在 open 段真正驱动筛选，不只当提醒文字：**
- **① discovery 候选**：偏亲子友好 / 低强度——避免长时间徒步、高强度项目、深夜场；优先可中途休息、室内、机动性强的点。
- **② planner 排点**：降单日强度——每天点位数下调、留缓冲、把累的点错开；长车程日拆短或当天加休整。
- **③ 跨城节奏**：缩转场频率——带娃 / 适老尽量少换城、少长途转场，能少跨就少跨。
- **④ 长车程日**：主动拆缓，不硬塞满。

例：案例1「-25℃ 带三年级娃跨双省」——不止「注意保暖」一句，而应：候选偏室内 / 亲子、单日强度下调、跨省段拆缓 + 加休整、避开极寒户外长时段。

**轻重控制（必守 · 判据是「会不会改变行程强度」，不是同行人数 / 代际数）**：触发本质是**同行里有老 / 幼 / 行动不便成员**才需降强度，不是「人多」或「几代人」。
- **普通成人结伴、情侣、朋友团、全员健康成人的多代同行（无幼童 / 无适老 / 无人行动不便）一律不触发**——别因「三代 / 多代」标签就无端降强度（误报）。
- **用户对强度的明确声明 > 同行结构的默认推断**：若用户带娃 / 适老但**明确点名**要高强度（带青少年徒步营、亲子滑雪进阶、长者本人坚持登山等），**不机械降强度抹掉该★必去**；当「降强度」与用户原话点名冲突时，按 9.0.7 轴2 摊牌回钩用户（保留高强度项 / 换低强度替代二选一），不静默砍。

**本条是半解（规则层 · 守零界面）**：让 Claude 在 open 段**主动**按同行结构降强度。让 `selectPagesForDay` / 界面 kids 字段**真正感知**同行结构并自动选页，是**缓做**（要碰 selector / 界面，不在本次）。

**⑥ 主题 / 朝圣 lens = 一等「加分」筛选 lens（正向驱动 · 病灶⑬ · 守零界面）**

> **病灶⑬**：约束清单 ①～⑤ 全是**减分 lens**（省钱→砍贵的、高反→砍高海拔、带娃→降强度），缺**加分 lens**——「打卡某剧 / 巡礼某地 / 朝圣某处」这类贯穿性主题偏好过去无结构位驱动选点，跨界面一稀释就丢，七天三城最后做成普通观光。预算 / 同行结构管「该不该排」（把关），主题管「优先排什么」（驱动）——两类 lens 语义相反、可并存、不互斥。

把用户**贯穿性**的主题 / 朝圣偏好（影视取景打卡、动漫圣地巡礼、文学 / 历史朝圣、演唱会 / 球赛追星等）写进 `meta.themes[]`（照搬 `meta.constraints` 的「Claude 消费的数据容器」模式，但**语义相反：constraints 减分、themes 加分**）。**单独建数组、不塞进 constraints**——避免主题被当成「减分过滤器」把非主题的吃 / 住 / 行候选全砍掉。结构：

```
meta.themes: [
  { name:'《苦尽柑来》影视朝圣', kind:'film', anchors:['取景地A','取景地B'], scope:'trip' },
  { name:'恶作剧之吻打卡',      kind:'film', anchors:['静宜大学','...'],   scope:'taichung' }
]
```

- `kind` ∈ `film`（影视）/ `anime`（动漫巡礼）/ `literary`（文学）/ `history`（历史朝圣）/ `music`（演唱会·追星）/ `sport` / `other`。
- `anchors[]`：该主题代表性点（取景地 / 朝圣地），含**用户原话点名的 ＋ Claude 主动 web_search 补齐的**同主题高价值点。
- `scope`：`'trip'` 全程驱动（如三城都打卡某剧）；填 region slug 则只在该城驱动（如恶作剧之吻只在台中）。

**关键 · 它是「加分排序 ＋ 锚点保入」lens，不是「减分过滤」lens（与 constraints 行为相反，务必分清）：**
- **① discovery 候选**：主题相关候选**顶到前排 / 给配额**，并**主动 web_search 扩展**同主题代表性点（不止用户点名的几个）。⚠ **绝不因此筛掉非主题的吃 / 住 / 行 / 常规观光候选**——主题是加法不是「只排主题点」，行程仍要完整、能吃能睡能移动。
- **② planner 排点**：把 `anchors` 当**优先锚点**先落位（像★必去一样保入），再围绕排周边；同主题点尽量串成动线、减少零散往返。**若某 anchor 同时被用户点名为★必去，合并为同一点、不重复呈现**（别既作★必去又作主题锚点出现两次）。**anchor 保入对账（必做 · 否则「保入」只是口号、病灶⑬复发）**：`meta.themes[].anchors` 里每个已过 9.0.0④ 核查、且在计划城市内的点，断裂 C 跑完必须能在某天 timeline 找到对应 card——缺一个即不通过，呼应 `city_plan.intercity` 的逐段对账。
- **③ scope 控制覆盖**：按 `scope` 决定全程驱动还是单城驱动，别把单城主题（恶作剧之吻@台中）误当全程 lens 拿去其他城找。

**与 9.0.0④ 具名实体核查衔接（必走 · 否则朝圣到错地方）**：`anchors` 里的具名取景地 / 朝圣点照样过 9.0.0④ 真实位置 / 可得性核查——**用户常记错剧名 / 取景地**（如「苦尽甘来」实为《苦尽柑来》、其釜山桥段实拍在陕川不在釜山）。主题负责「优先排」，9.0.0 把关「排得对不对」，互补不可省。

**渲染出口（必做 · 否则用户看不到主题，病灶⑬等于没修）**：`meta.themes[]` 自身**不出图**（模板只渲染 `meta.constraints`；themes 同 `city_plan` 是 Claude 消费的骨架）。断裂 C 组装 TRIP 时**必须**把每个 theme 转写成一条 `meta.constraints`（`type:'theme'`、**scope 原样沿用 `themes[].scope`**——单城主题误写 `'trip'` 会让渲染层漏标【region】前缀、用户误以为全程都打卡；text 如『🎬 本趟贯穿《苦尽柑来》影视朝圣，已优先安排取景地』，**带 🎬 前缀**让它在 other 兜底分组里仍一眼可辨），由「出行须知」区统一渲染（已验证唯一保证不丢的出口）。`type:'theme'` 当前经「其他 / 📌」兜底渲染、不丢；专属图标分组（模板 `CSTR_TYPE` 加 `theme` 键 + `CSTR_ORDER` 补位，**纯模板改动**）属缓做。**只填 `meta.themes` 不转写 = 用户看不到 = 病灶⑬没修好**（与 9.0.6 `border_notes`→`constraints` 同一转写模式）。

**轻重控制（必守 · 防硬造主题 + 防过度盘问）**：触发本质是用户表达了**贯穿性主题主线**（点名某剧 / 某主题要打卡 / 朝圣 / 巡礼）。零散一句「顺便看看 X」不升 theme；普通观光无明确主线**不硬造主题、不追问「你有没有什么想打卡的」**。主题型偏好与负向约束并存正常（背包客省钱 ＋ 影视朝圣同时成立，分别落 themes / constraints）。**升级阈值（粗略）**：同主题 anchor 仅 1 个孤立点 → 倾向**不升 theme**、按普通★必去处理即可；≥2 个同主题点、或用户明说「巡礼 / 朝圣 / 多去几个剧里的地方」→ 升 theme 当贯穿 lens。

**与 9.0.7 衔接**：当某主题 `anchor` 落在计划城市 / 区域外、或本地稀缺（9.0.0④ 查出），是否专程去 = 影响全局的关键岔路，交 9.0.7 摊牌门回钩用户（专程一日游 / 放弃 / 顺路替代），不静默砍也不静默硬塞。**同一 theme 多个 anchors 同时落在计划外时，聚合成一次摊牌**（『这部剧核心取景地其实散在 X / Y / Z，都不在你计划的城市——要不要调整城市组合 / 取舍哪几个 / 只打卡能顺路的部分？』），别逐点碎问。例：《苦尽柑来》核心村景是安东（庆北）搭的布景、初吻在高敞（全北），济州只有部分海景/实景，用户若按「济州朝圣」排满三城会扑空。

**本条定位**：纯规则层（守零界面）。让 Claude 在 open 段把主题当加分 lens 主动驱动选点 ＋ 锚点保入；`anchors` 作普通 `sight` / `experience` 卡正常出页，不依赖任何新界面字段，故无 selector / 界面缓做项。

#### 9.0.6 多城城际骨架捕获（病灶⑤修复）

> **病灶⑤修复**：「城市顺序 + 每城待几天 + 城际交通方式」这套多城前置决策过去散在 Stage 0 对话和 medium-multi 口头规则里，没有结构化产物，下游 planner / 断裂 C 无法消费，多段城际容易漏造。本节强制在 Stage 0 固化进 `meta.city_plan`。

**① 触发**：目的地 ≥2 城（呼应 Archetype 的 medium-multi / long）。单城跳过本节。多城（≥2 城）行程未产出 `meta.city_plan` = P0 流程违规，城际对账直接判 fail——锚缺失本身就是失败条件，不得静默跳过。

> **跨境额外触发（病灶⑫修复）**：凡**出发国 ≠ 目的国**，或**经第三地中转往返**（如深圳经香港往返台湾），无论目的地几城，都额外触发 `entry_exit` 国际出入境段捕获（见②结构里的 `entry_exit`）。**单城但跨境**也要建 `meta.city_plan`（`cities` 只一项、`intercity` 留空），仅为承载 `entry_exit` 入境段——否则跨境提醒无处可挂、必丢。

**② Stage 0 末尾必做 · 结构化捕获**：产出 `meta.city_plan`，含城市顺序 + 每城天数 + 城际交通段，作为后续 planner 注入分天、断裂 C 合成转场段的骨架锚。结构：

```
meta.city_plan: {
  cities:   [ { name:'曼谷', region:'bangkok', days:3 }, { name:'清迈', region:'chiangmai', days:2 }, { name:'普吉', region:'phuket', days:2 } ],  // 按游览先后排序
  intercity:[ { from:'曼谷', to:'清迈', mode:'flight', duration:'约1.5h' }, { from:'清迈', to:'普吉', mode:'flight', duration:'约2h' } ],      // 相邻两城各一段
  // ↓ entry_exit：仅跨境行程才有。出发国→目的国的「国际出入境段」既不是目的地 city、也不是两 city 间的 intercity，过去无处可放（病灶⑫）
  entry_exit: {
    inbound:  { from:'深圳', gateway:'香港', to:'台中', mode:'flight', note:'深圳经港中转入境' },  // 出发地 →（中转口岸 gateway）→ 目的国首站
    outbound: { from:'台北', gateway:'香港', to:'深圳', mode:'flight' },                          // 目的国末站 →（中转口岸 gateway）→ 出发地
    border_notes: [ '入台证/签注正本随身+电子档备份', '海关申报与禁带物', '落地电话卡/网卡', '口岸·机场→市区交通', '经第三地中转：行李是否直挂、是否需过境签' ]  // 跨境「落地操作」提醒挂载位（有效期类归 9.0.8，不在此重复）
  }
}
```

> **`entry_exit` 与 9.0.8 分工**：9.0.8 资格准入门管「**有没有资格**跨境」（签证 / 免签有效期 / 护照 ≥6 月）；这里的 `entry_exit` 管「**怎么落地**」的操作层（中转口岸、海关申报、落地电话卡、口岸→市区交通、行李直挂 / 过境签）——是入境段在城际骨架里的**结构承载位**。**证件有效期（签证 / 入台证 / 护照 ≥6 月）一律归 9.0.8 核查，`border_notes` 只装落地后的操作动作，不重复写有效期。** 两者互补不重复。
>
> **⚠ 渲染出口（必做 · 否则等于没修，病灶⑫只是从「无处存」挪成「存了不出图」）**：`entry_exit` 自身**不出图**（模板只渲染 `meta.constraints`，`city_plan` 是 Claude 消费的骨架、不进出图）。断裂 C 组装 TRIP 时**必须**把 `border_notes` 每条转写成一条 `meta.constraints`（`type:'transport'`、`scope:'trip'`），由「出行须知」区统一渲染（已验证的唯一保证不丢的出口）；`inbound`/`outbound` 的「口岸·机场→市区交通」落进抵达 / 离境当天的 T2 转场页。**只填进 `entry_exit` 不转写 = 用户看不到 = 病灶⑫没修好。**
>
> **中转口岸 `gateway` 不是目的地**：经第三地中转的口岸（如深圳经港的「香港」）只作出入境段的**途经点**，**不建 region、不参与 region 候选配额、不单独占游览天**（收编原 TRAVEL-006「香港中转无 region 归属」）。除非用户明确要在中转地停留游玩，那才升级为一座正式 `cities[]`。

**③ 天数对账**：各城 `days` 之和（含转场日归属）= 总天数；对不上即修正。

> **过夜驻点必登记进 `cities[]`（病灶⑮ · 自驾 / 多点自然环线最常踩）**：过夜的驻点即使是**景区村 / 自然驻点**（如布尔津、禾木、贾登峪、白哈巴），而非行政市，也**必须登记为一条 `cities[]` 条目**——`name` 填村 / 驻点名、`region` 给驻点级 slug（如 `hemu` / `jiadengyu`）、`days` 填在该点的实际驻留天数。原因：天数对账靠 `Σcities[].days == 总天数` 成立；若过夜的景区村不登记，那几天**没有归属对象**，加总必然对不上、且选页器按天聚焦时无城可锚。**判据**：只要在某点「过夜」就要建 `cities[]` 条目，不论它是不是行政市；纯白天路过不过夜的点不建城（按工序③.5 当沿途景观点处理即可）。
>
> **⚠ 与城际段对账的口径统一（病灶⑮ 补 · 防自驾环线连续过夜驻点被误判 fail）**：自驾环线里「布尔津→禾木→贾登峪」这种**连续自驾过夜的景区驻点**虽都进 `cities[]`，但村间路程属「**同一自驾段内的连续停留**」，**不**按「相邻城各造一段独立 intercity transit」处理，**也不计入 9.2 的 intercity 段数对账**——否则会和「沿途点当 landing 点、不算城际」的修法自相打架，且让 9.2 段数凭空多出几段、对不上误判 fail。**判定边界（与工序③.5 沿途点口径统一）**：自驾连程里**白天路过、不过夜的点 = 沿途 landing 点**（不进 cities[]、不算城际段）；**晚上过夜的驻点 = `cities[]` 条目**（占 days、参与天数对账，但村间连程仍不算独立 intercity 段）。二者不冲突：过夜=建城条目、白天路过=沿途点，按是否过夜一刀切。**⚠ 但「过夜→过夜」的自驾转场日仍要出 T2**：两个过夜驻点之间的自驾转场日（如禾木过夜→次日开车到贾登峪过夜），那天 timeline **仍必须造一条 `{type:'transit',intent:'intercity'}`**——选页器是**按天**靠 `intent:'intercity'` 才出 T2 转场页，不造会被渲染成「城内闲逛页」（开了一整天车却出错页型）。这里说的「不算独立 intercity 段」只指**不登记进 `city_plan.intercity`、不计入 9.2 段数对账**，timeline 上那条 transit 仍要有（只为驱动当天出 T2）。区别于「**同一天**连续路过多个不过夜的点」——那种当天连程才**完全不造段**、沿途点按工序③.5 当 landing 处理。

> **转场日天数归属口径（两种分支 · 病灶⑫修复补②）：**
>
> **① 常规当日转场**（车程 / 航班当天就到）：转场日**归入出发城**——那天上午在出发城活动、下午搭车赶路，主活动归出发城；到达城当天傍晚才到、不单独占一天，从转场日次日起算。各城 `days` 严格不重叠，之和 = 总天数。（与 9.1 转场日活动分配规则自洽：出发城活动只放赶车前、到达城活动只放抵达后。）
>
> **② 跨夜长途转场**（夜船 / 红眼航班 / 夜行卧铺，整段**跨过午夜**）：这一「在途夜」**单独占一天、作独立 T2 赶路日，既不归出发城也不归到达城**。口径：出发城活动算到登船 / 起飞当天的尾巴，到达城从抵达**次日**起算，中间这个「跨午夜在途日」单列对账。原「归出发城、到达城不占天」的口径在此场景**失效，不可套用**。例：案例3 济州夜船晚 7 开、次日早 6 到，整晚在船跨午夜——这一天是独立赶路日，不算进济州也不算进出发城。
>
> **断裂 C 生成（必做 · 否则「单独占一天」只是空口径没人执行）**：为这个跨午夜在途夜单独造一个 `day` 对象（`theme` 如『夜船 首尔→济州 · 跨夜赶路』），其 `timeline` 含该 `{type:'transit',intent:'intercity'}` 段 **＋ 一个 `type:'free'` 锚点**（如『船上过夜 · 跨夜赶路』）。⚠ **必须带这个 free 锚**：`selectPagesForDay` 有个 `hasSubstance` 门，纯 transit、无任何 point/free 的 day 会被判成 **T8 休整页、不是 T2 赶路页**；`free` 在选页器里算 point，带上它才稳出 T2。到达城首日 `day` 从抵达**次日**另起。
>
> **跨夜分支下的天数对账**：`各城 days 之和 + 独立跨夜赶路日数 + entry_exit 独占天 = 总天数`。其中 **entry_exit 独占天**仅当国际出入境段无法被首尾城当天吸收、需单独占一天才计入（如返程经第三地转机过夜）；多数当天到 / 当天走的国际段被首尾城吸收、不另占天（如案例2 深圳经港当天到台中，国际段算进台中 day1）。判定「跨夜」只看交通段是否真跨当地午夜（夜船 / 红眼）；当天到的常规转场仍走分支①，不要把普通晚班车误判成独立赶路日（防过度设计）。
>
> **交通段真实性归上游门**：`intercity.mode` 标注的夜船 / 航线**是否真实存在且在运营**，归 9.0.0 事实预检 / 9.0.7 可行性摊牌核实（9.0.6 只负责结构承载，不背书航线存在）。例：「首尔→济州夜船」现实里夜船口岸在釜山、且该航线可能已停运——这类要 9.0.0/9.0.7 拦下摊给用户，不能静默排进 timeline。

**④ discovery 候选配额**：候选量按各城天数比例配额（discovery 原生支持多城 `cities[]` tab + 按 `cityId` 配候选，纯注入控制，**不改界面**）。

**⑤ 不另造分图/分页机制（铁律）**：planner 多城用**单个文件**，按天聚焦当前城（planner 地图按当前天点位自动 fit，不绑定目的地中心）；攻略页**按天分页、每天归属其城、城际日出 T2**——跨城游产出粒度就是「天」，由现有「按天出页」原生覆盖，**不拆 planner 文件、不分图、不分次跑**。一天只在一个城活动，城际移动是「转场日」那一天的事。

#### 9.0.7 整趟可行性摊牌门（病灶⑦修复 · 离开 Stage 0 triage、进研究/排点前的最后一关）

> **病灶⑦修复**：skill 过去所有校验都是**局部**的——9.0.0 查前提（季节/地理/玩法对不对）、9.1 查当天 region、9.2 查城际段数——却缺一道「把整趟所有硬诉求摆一桌、看它们在给定天数/预算/地理尺度下能不能**同时成立**」的宏观关。结果互相打架的诉求被闷头硬排或静默砍掉（7 天 3 地含远端硬塞、15 天又要宽松又要逛遍、1w5 背包档却点名野奢营地、关键岔路替用户默默选），用户拿到成品才发现不对。本门强制在动手研究/排点前，先把矛盾摊给用户裁决。

**触发时机**：9.0.1 分诊 + 9.0.5 约束 + 9.0.6 城际骨架捕获完成后、进入 discovery / planner / 深入研究之前——这是离开 Stage 0 triage 的最后一道关。

**四个交叉检查轴（逐项过，命中冲突即摊牌）：**

1. **地理跨度 ÷ 天数**：把所有必去点 / 必去城摆上地图，估首尾或对角最大跨度 + 城际交通总耗时占总天数的比例。交通吃掉过多天数（经验阈值 **>30%**）、或必去点在给定天数 + 交通方式下兜不拢 → **冲突**。例：案例2 阿勒泰（最北）+ 沙漠营地（东南）对角 1000km+、15 天兜不过来。（承接 9.0.0 查项①标记的「同一目的地内多个必去点相距过远」，把"标记"升级为"逼摊牌"。）
2. **预算档 vs 点名高价必去项**：`meta.budget_tier` 要筛掉的档次里，有没有用户**原话点名**的高价必去体验（即 §I-1 锁定的 ★必去）？有 → **冲突**，不静默砍。例：案例2 1w5（背包客档）vs 点名的「沙漠野奢营地」；案例4「每人 1w5 + 必要时背包客 + 一半住青旅」混合弹性预算无对账。
3. **节奏诉求 vs 数量诉求**：用户既要「宽松 / 不赶」又要「尽量多看 / 塞 N 个地方」？→ **冲突**。例：案例2「15 天宽松」vs「尽量把新疆好看的都去一遍」。
4. **关键未定决策岔路**：用户明说「方案没想好」、但存在影响**全局**的关键岔路（交通方式 / 城市取舍 / 进出点）→ 主动把岔路 + 各选项利弊提出来让用户拍板，**不替他默默选**。例：案例3 济州岛飞机（55 分）vs 夜船（11-12h）。

**轻重控制（必守 · 防过度设计）**：多数行程诉求本就自洽，**不要硬找矛盾**。只有某一轴「摆到一起明显兜不过来 / 打架」才摊牌；四轴都顺 → 一行带过「诉求自洽，进研究/排点」即可。本门目的是**省用户返工**，不是制造决策负担。

**产出与回钩**：命中冲突 → **不闷头硬排、不静默砍诉求**，列「你说的几个诉求 + 为什么打架」+ **2 个具体取舍方案**回钩用户裁决（复用 §I-1 消歧回钩机制：列候选 + 一句区别理由，等用户拍板）。例：「15 天又要宽松又要把新疆逛遍——兜不到一起。方案A：只深度玩北疆一个区（喀纳斯+禾木+赛里木湖）、宽松；方案B：15 天紧凑环线尽量多看、但天天赶。你选哪个？」**用户裁决后才进研究/排点。**

**固定输出格式（参照 9.0.0 的 `【事实预检】` 代码块风格）：**

```
【可行性摊牌】（离开 triage · 进研究/排点前）
- 地理×天数：✓ / ⚠ ____与____相距约__km，__天兜不过来 → 方案A__ / 方案B__
- 预算×诉求：✓ / ⚠ ____档筛掉了你点名的____ → 保留(超预算约__)/换平价替代？
- 节奏×数量：✓ / ⚠ 又要"____"又要"____"，冲突 → 方案A宽松少点 / 方案B紧凑多点
- 关键岔路：✓ / ⚠ ____没定、影响全局 → 选项1__ / 选项2__
```

四轴都顺则一行「诉求自洽 ✓，进研究/排点」。

**与其他门的分工**：9.0.0 查「前提对不对」（facts）、9.0.7 查「诉求能不能同时成立」（feasibility）、9.0.3 查「路径判轻没 / 该不该减法」（triage）。三者互补、不重复。

#### 9.0.8 出行资格准入门（病灶⑪修复 · 查"人有没有资格用这种方式出行"）

> **病灶⑪修复**：skill 过去的核查全是「地点能不能去」（开放 / 季节 / 坐标），从不查「出行主体有没有资格用这种方式去」。结果自驾攻略做完了，落地才发现驾照不被承认、租不到车、不能合法上路；或跨境免签政策已过期，出行年份理解错就白做一趟。本门在 Stage 0 按**出行方式**触发资格核查，把「人能不能合法用这种方式出行」这道准入门补在动手研究前。

**触发**：仅当行程涉及以下任一才启动深查（否则一行带过「无特殊资格门槛 ✓」）——① 自驾 / 租车 ② 跨境出境 ③ 需资质的特殊活动（潜水 / 入山徒步 / 特种驾驶等）。常规跟团 / 公共交通 / 不跨境的国内游不触发，不要无中生有找门槛。

**三类核查清单（命中即真实 WebSearch 核实目的地的具体要求，不凭印象）：**

1. **自驾 / 租车资格**：目的地是否承认用户的驾照？（**注意：中国大陆驾照未加入国际驾照公约、各国认可规则不一，多数需「驾照 + 公证认证翻译件」或当地特定认证，如澳洲 NSW 需 NAATI 认证英文翻译件**）+ 租车最低年龄 / 驾龄 / 信用卡要求。命中限制 → 回钩告知 + 给替代（出发前办认证翻译件 / 国际驾照 / 改包车 / 当地参团一日游）。

> **长途自驾能力边界告知（病灶⑮ · 必做 · 防用户期待落空）**：当判定为**长途自驾**时，必须主动一句话告知用户当前模型的局限——「当前按**离散点位 + 公共交通班次模型**建模，自驾沿途景观会简化为停留点，**不含里程 / 油费 / 路况 / 连续路线**，长途环线的连续驾驶体验会打折」。**判据（命中任一即算长途自驾）**：① 单段自驾转场里程大（经验阈值 >150km / 单段）；② 自驾天数占总天数比例高（>50%）**且**总自驾里程达长途量级（经验阈值 >1000km）或跨地区 / 跨气候带（加里程 / 跨区限定，单防「天数占比高但其实是双城短途」误命中）；③ 过夜点多为景区村而非城（呼应③的 `cities[]` 驻点登记）。**轻重控制（必守 · 防过度打扰 · 防过度设计）**：**「双城短途自驾不触发」豁免凌驾于以上任一单条判据之上**——`双城 / 单跨短途自驾即使自驾天数占比高（如 3 天双城两天在开车 = 66%），只要总里程未达长途量级、未跨大区，就不触发`，现有离散点位 + 班次模型对短途双城已验证够用，不要为它加这段告知。判定顺序：**先过豁免（是不是双城 / 单跨短途）→ 是则直接不触发，不再逐条看判据；否则才按①②③判**。只有命中长途判据**且不落在豁免里**的环线才告知。
2. **跨境签证 / 入境许可**：签证或免签 / 电子签要求 + **有效期**（如 K-ETA 豁免有截止日，须确认出行年份在不在豁免期内）+ 护照有效期（多数国家要求 ≥6 个月）+ 入境特殊要求（往返票 / 资金证明 / 疫苗等）。
3. **特殊活动资质**：潜水（执照 / 体检）、入山徒步（许可证 / 强制向导）、特种驾驶 / 狩猎等需许可的活动。命中 → 告知所需资质 + 办理路径。

**轻重控制（必守 · 防过度设计）**：常规跟团 / 公共交通 / 不跨境的国内游，无需逐项查，一行带过「无特殊资格门槛 ✓」。只有命中自驾 / 跨境 / 特殊活动才深查。**具体国别规则一律 WebSearch 现查、不在规则里写死**（政策易过时，写死必踩坑）。

**产出与回钩**：命中资格门槛 → 提前告知用户 + 给「补办路径」或「替代方案」，并落进 `meta.constraints`（自驾 / 签证类用 `type:'transport'`，其他用 `type:'other'`，`scope` 按适用范围填 `trip` / `day:N`；复用 §I-1 回钩机制：列要点 + 一句理由等用户确认）。例（自驾）：「中国驾照在澳洲 NSW 上路需 NAATI 认证英文翻译件，否则租不到车。建议：① 出发前办好 NAATI 翻译件；② 或改包车 / 当地一日游团。」例（签证）：「香港居民免 K-ETA 政策截止 2025-12-31——确认你的出行是 2025 国庆（在豁免期内）还是 2026（需重新申请 K-ETA）。」

**固定输出格式（参照 9.0.0 的代码块风格）：**

```
【资格准入】（仅当涉及自驾 / 跨境 / 特殊活动）
- 自驾资格：✓ / ⚠ ____驾照在____需____（认证翻译件/IDP）→ 出发前办____ 或 改包车
- 签证入境：✓ / ⚠ ____（签证/免签有效期/护照6个月）→ ____
- 特殊活动：✓ / ⚠ ____需____（执照/许可/向导）→ ____
```

不涉及自驾 / 跨境 / 特殊活动则不输出此块（或一行「无特殊资格门槛 ✓」）。

**与其他门的分工**：9.0.0 查**前提事实**（季节 / 地理 / 玩法 / 具名点对不对）、9.0.7 查**诉求可行**（诉求能不能同时成立）、**9.0.8 查主体资格**（人能不能合法用这种方式出行），9.1+ 查数据结构。四者互补、不重复。

### 9.2 地图坐标密度（M 规则）

**写完 cards 数据后，运行下面核查：**

```js
// 快速自检：每天有效坐标点数
trip.days.forEach(d => {
  const pts = d.timeline.filter(t => t.card_id && trip.cards[t.card_id]?.coords);
  if (pts.length < 3) console.log(`⚠ D${d.num} 坐标点只有 ${pts.length} 个`);
});
// 检查 sight/meal 卡片是否有 coords
Object.entries(trip.cards).forEach(([id, c]) => {
  if (['sight','experience','meal_breakfast','meal_lunch','meal_dinner','meal_snack'].includes(c.type) && !c.coords) {
    console.log(`⚠ 缺坐标: ${id} (${c.name})`);
  }
});
```

- [ ] 所有 sight/experience/meal 类型 card 有 coords
- [ ] 每天 timeline 中带 coords 的点 ≥ 3（不足则补锚点卡片：酒店/车站/广场）
- [ ] 跨城转场日至少有出发地和目的地两个坐标锚点
- [ ] **城际段数对账**（多城 ≥2 城）：`{type:'transit',intent:'intercity'}` 段数 == `meta.city_plan.intercity` 段数，逐段比对 from/to，缺即 fail（防三城两段漏造中间一段）。**自驾环线豁免（病灶⑮ 补）**：本条「cities 相邻必造 intercity」硬要求对**自驾环线不成立**——自驾环线的城际 / 跨区段数，以**实际的公共交通班次段（飞机 / 火车 / 长途客运）或长途跨大区转场**为准；连续自驾过夜驻点的**村间自驾连程不计入本段数对账**——`meta.city_plan.intercity` 在自驾环线里**只登记真正的公交班次段 / 跨大区转场**。**注意区分两种情形**（详见 9.0.6 口径统一）：① **同一天**路过多个不过夜的点（如一天内「布尔津→五彩滩→…」）→ **完全不另造 intercity 段**、沿途点按工序③.5 当 landing 处理；② **过夜→过夜的自驾转场日**（如禾木→次日贾登峪）→ 那天 timeline **仍要造 `{type:'transit',intent:'intercity'}`** 让选页器出 T2，但该段**不登记进 `city_plan.intercity`、不计入本对账**。所以本对账只逐段比对 `city_plan.intercity` 登记的公交 / 跨区段，自驾转场日为出 T2 而造的 intercity 段**不计入、不因此判 fail**。
- [ ] **城际段距离合理性复核**（多城 ≥2 城 · 9.0.0 的最后防线）：反常识超长城际段（跨大区 / 跨气候带，如"吉林→腾冲 3000km"）本应已被 9.0.0 事实预检拦截；此处作为最后防线，复核每条 `{type:'transit',intent:'intercity'}` 段的距离 / duration 与 mode 是否合理。命中反常识段 → **回 9.0.0 事实预检消歧（列「用户说的 vs 核实到的」回钩用户），不静默通过**
- [ ] **自驾日沿途点防丢对账**（病灶⑮ · 凡含自驾 / 自驾环线转场日必做）：逐个自驾转场日核对——该日 timeline 里实际造出的沿途景观点（`sight` / `sight_nature`）数量 == 行程实际经过的沿途点数，**且这些点全部排在该日所有 transit 段之后**（不得夹在多段 transit 之间）。原因：选页器落地点提取是 `landing = timeline.slice(lastCrossIdx+1)`，只取「末段跨城 transit 之后」的点，夹在中间的会被 slice 静默切掉、用户彻底看不到。命中「点数对不上」或「点夹在 transit 中间」→ fail，按 SKILL 断裂 C 工序③.5 重排该日 timeline（先连排 transit，再排沿途点）
